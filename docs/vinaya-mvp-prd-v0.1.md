# Vinaya MVP 产品需求文档 v0.1

## 一、文档目标

本文档用于定义 Vinaya 第一版最小可行产品（MVP）的范围、目标用户、核心流程、功能边界与验收标准。Vinaya 的第一版目标不是做成一个复杂的多 Agent 平台，而是验证“先净化判断，再允许执行”的核心价值是否成立。

## 二、产品定位

Vinaya 是一个判断净化系统，用于在 AI 输出进入现实执行前，增加一层发心、观缘、持戒、辩义、缓行、回向的流程控制。

一句话定位：

`Vinaya：不是让 AI 更快决定，而是让 AI 更少犯业。`

## 三、MVP 目标

MVP 只验证四件事：

- 用户能提交一个高影响判断请求
- 系统能跑完整个净化流程并给出三类结论
- 系统能记录完整因果簿
- 用户能在界面中查看结果并进行人工复核

## 四、非目标

第一版明确不做：

- 多租户与复杂权限体系
- 外部平台深度集成
- 自动执行大量真实业务动作
- 自学习和自动改写规则
- 复杂消息编排与多团队协作
- 高保真宗教风视觉系统

## 五、目标用户

### 1. 产品设计者

需要一个可控的 AI 决策前置层，用于高风险功能上线前的判断净化。

### 2. 运营或审核人员

需要对高影响请求做辅助判断，但不希望 AI 直接越权裁决。

### 3. 技术团队

需要一个可审计、可回退、可人工接管的决策中间层原型。

## 六、核心使用场景

### 场景 A：内容治理建议

用户提交“是否应该对某用户做封禁”的请求，系统输出：

- 发心分析
- 因缘分析
- 持戒警告
- 更低伤害替代方案
- 最终结论：可行 / 缓行 / 止行
- 人工复核建议

### 场景 B：高风险功能开关建议

用户提交“是否开放某自动审核功能”的请求，系统判断是否应先试点、小范围灰度或直接止行。

### 场景 C：内部制度建议

用户提交“是否采用某一项更严格的团队策略”的请求，系统分析代价承担者、潜在误伤、执行边界和缓行路径。

## 七、核心用户流程

MVP 的标准流程如下：

1. 用户在 Web 页面填写请求内容
2. 系统生成结构化请求对象
3. 系统串行运行五个净化处理器：
   - 发心
   - 观缘
   - 持戒
   - 辩义
   - 缓行
4. 系统汇总结果，生成最终结论：
   - 可行
   - 缓行
   - 止行
5. 系统写入因果簿
6. 用户在详情页查看判断链
7. 用户可提交人工复核意见
8. 系统记录复核与后续状态

## 八、核心功能列表

### 1. 请求提交

功能说明：

- 提供一个表单提交判断请求
- 输入字段最少包括标题、请求内容、业务领域、风险等级、补充上下文

验收标准：

- 用户可成功提交请求
- 请求有唯一 ID
- 请求数据可被后续流程读取

### 2. 请求标准化

功能说明：

- 将用户输入转为统一结构
- 自动清洗空白、无效字段和基础元数据

验收标准：

- 所有请求都生成标准 JSON 对象
- 缺失字段有默认值或错误提示

### 3. 发心分析

功能说明：

- 识别主要动机、受益方、代价承担者与混杂动机

验收标准：

- 输出至少包含 primaryIntent、beneficiaries、costBearers、mixedMotives

### 4. 观缘分析

功能说明：

- 识别近因、远因、深层因、外溢后果和影响对象

验收标准：

- 输出至少包含 proximateCauses、rootCauses、affectedParties、externalities

### 5. 持戒检查

功能说明：

- 基于 Vinaya 五戒做规则检查
- 标记 warning 或 hard block

验收标准：

- 可输出戒律检查结果列表
- 发现高风险时可触发人工复核标记

### 6. 辩义建议

功能说明：

- 生成多个备选路径
- 给出更低伤害、更可回退的优先方案

验收标准：

- 至少输出 2 个备选方案
- 明确 preferredOption
- 记录一条 dissentNotes

### 7. 缓行方案生成

功能说明：

- 对中高风险请求生成试行计划
- 明确观察期、回退条件、人工负责人

验收标准：

- 输出 mode
- 输出 trialPlan
- 可标记 rollbackEnabled

### 8. 最终结论生成

功能说明：

- 汇总前述所有阶段，输出三类结论之一

验收标准：

- 只允许出现 `allow`、`defer`、`stop`
- 必须附带 reasoningSummary

### 9. 因果簿

功能说明：

- 保存每次请求的完整判断链
- 可在详情页查看

验收标准：

- 每个请求均自动生成 ledger 记录
- 详情页可查看各阶段输出

### 10. 人工复核

功能说明：

- 用户可对结果进行人工复核
- 可记录“维持、修改、推翻”三类复核结果

验收标准：

- 系统支持提交复核意见
- 复核结果可写入审计记录

## 九、页面范围

MVP 只做 4 个页面：

### 1. 首页 / 请求提交页

内容：

- 项目说明
- 请求表单
- 最近请求入口

### 2. 请求列表页

内容：

- 请求标题
- 风险等级
- 当前状态
- 最终结论
- 创建时间

### 3. 请求详情页

内容：

- 原始请求
- 五阶段输出
- 最终结论
- 因果簿
- 人工复核入口

### 4. 规则说明页

内容：

- Vinaya 五戒
- 三类结论说明
- 缓行原则
- 人工主权边界

## 十一、状态设计

建议 MVP 状态：

- submitted
- normalized
- intention_reviewed
- causality_reviewed
- precepts_reviewed
- deliberated
- deferred
- allowed
- stopped
- reviewed
- archived

## 十二、技术建议

为了尽快落地，MVP 建议：

- 前端：Next.js
- 后端：Next.js Route Handlers 或独立 API
- 存储：SQLite 或 PostgreSQL
- ORM：Prisma
- 样式：Tailwind CSS
- LLM 访问：统一封装 provider client

如果要更快，也可以第一阶段先用 mock 处理器，不立即接真实模型。

## 十三、验收标准

MVP 完成的最低标准为：

- 可以提交请求并落库
- 可以运行完整五阶段判断
- 可以输出 allow / defer / stop
- 可以在详情页查看完整判断链
- 可以提交人工复核
- 可以查看因果簿摘要

## 十四、版本边界

v0.1 只证明“判断净化流程成立”。

它不证明：

- 多 Agent 协作复杂度
- 实时调度能力
- 高并发能力
- 企业级权限控制
- 复杂自动化执行

这些属于后续版本。

## 十五、成功指标

MVP 阶段重点看：

- 用户是否理解三类结论
- 缓行是否真的成为常见输出
- 人工复核是否认为结果有帮助
- 因果簿是否能支持复盘
- 产品是否清楚地不同于普通 Agent 工作流

一句话，MVP 的成功不是“做得多”，而是“把 Vinaya 的气质做出来”。
