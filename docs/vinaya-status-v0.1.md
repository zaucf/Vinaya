# Vinaya 项目现状总结 v0.1

> 最后更新：2026-03-16

本文档对照项目宪章、架构设计和 MVP PRD，逐项审视当前实现与设计初衷的一致性，记录已完成功能、偏离点和后续方向。

---

## 一、设计初衷与当前实现对照

### 1. 核心定位：是否守住了？

宪章定义：
> Vinaya 是一个以戒、定、慧为核心的 AI 判断净化系统。它的职责是在"请求"与"执行"之间插入一层净化流程。

其中：
- **戒**（持戒）——五戒红线，为判断划定不可逾越的底线
- **定**（克制/缓行）——不急于下结论，"缓行"是成熟系统的能力而非失败
- **慧**（判断净化）——六阶段分析、因果记录、持续修正

**结论：守住了。**

当前系统的核心链路就是：用户提交判断请求 → LLM 引擎完成六阶段净化分析 → 输出 allow / defer / stop 三类建议 → 人工复核 → 因果簿记录。没有跑偏去做任务调度、自动执行或多 Agent 编排。系统始终围绕"判断净化"这一件事运转。

### 2. 六阶段流程：是否完整？

| 阶段 | 宪章定义 | 当前实现 | 状态 |
|------|---------|---------|------|
| 发心 | 检视动机、受益方、代价方、杂染动机 | LLM 输出 primaryIntent / mixedMotives / beneficiaries / costBearers / intentionRisk | ✅ 完整 |
| 观缘 | 分析近因、远因、影响链、外溢后果 | LLM 输出 proximateCauses / rootCauses / affectedParties / externalities / causalityRisk | ✅ 完整 |
| 持戒 | 五戒硬约束、越戒标签、止行/人工升级 | LLM 输出 preceptFindings / hardBlock / humanReviewRequired / preceptRisk | ✅ 完整 |
| 辩义 | 多视角辩论、替代方案、最低伤害路径 | LLM 输出 options / preferredOption / dissentNotes / deliberationRisk | ✅ 完整 |
| 缓行 | 分阶段执行、限权、可回退策略 | LLM 输出 mode / trialPlan / releaseRisk | ✅ 完整 |
| 回向 | 记录因果链、误伤、修订建议 | 因果簿页面展示完整判断链；人工复核可记录维持/修改/推翻 | ⚠️ 部分（缺少自动误伤检测和规则修订建议） |

**结论：六阶段中前五阶段完整实现，第六阶段（回向）的因果记录部分已实现，但补赎机制和自动规则修订尚未落地。**

### 3. 五戒：是否生效？

| 戒律 | 宪章定义 | 当前实现 | 状态 |
|------|---------|---------|------|
| 不妄语 | 不编造事实、不伪装确定性 | LLM 系统提示词约束 + 持戒引擎检查 + 规则中心可配置 | ✅ |
| 不害生 | 不为效率制造伤害 | 同上 | ✅ |
| 不偷夺 | 不剥夺机会和申诉空间 | 同上 | ✅ |
| 不越界 | 不超出授权边界 | 同上 | ✅ |
| 不昏乱 | 证据不足不做强判 | 同上 | ✅ |

五戒已内嵌于 LLM 系统提示词中，持戒引擎会对每条戒律输出 pass / warning / block 状态。规则中心页面（`/rules`）支持在线启用/禁用每条戒律、切换严重等级（warning / block）。

**结论：五戒框架完整落地。但当前依赖 LLM 自觉遵守提示词，没有独立于 LLM 的硬规则校验层。**

### 4. 三类结论：是否严格？

宪章要求只输出 allow / defer / stop。

当前实现中，`llm_pipeline.py` 和 `pipeline.py` 的 `decision` 字段只允许这三个值，类型定义 `Decision = Literal["allow", "defer", "stop"]` 在编译期强制约束。前端展示也严格对应三种颜色标签。

**结论：严格一致。**

### 5. 人工主权原则：是否保留？

宪章要求：最终伦理责任必须属于人。

当前实现：
- 高风险请求 LLM 默认输出 `humanReviewRequired: true`
- 详情页有人工复核入口，支持"维持、修改、推翻"三类结论
- 复核记录支持多条历史，按时间倒序展示
- 规则中心可配置"强制人工复核最低风险等级"

**结论：人工主权入口已建立。但尚未实现"止行时必须人工确认才能归档"等强制流转约束。**

## 六、主要偏离与简化

### 6.1 状态机未显式实现

架构设计了 13 个状态（submitted → normalized → intention_reviewed → ... → archived），但当前实现是 LLM 一次调用串行完成所有阶段，没有显式的状态流转和持久化中间状态。

**影响**：无法观察到请求"正在哪个阶段处理中"，也无法在中途暂停或干预。对 MVP 阶段影响不大，但后续支持长时间任务或异步处理时需要补上。

### 6.2 辩义引擎未拆分为多角色

架构设计了观者、戒者、悲者、行者、记者五个系统角色，各自独立分析。当前由单次 LLM 调用统一处理。

**影响**：输出质量依赖 LLM 的提示词理解力，缺少多视角的真正对抗和分歧。后续可以考虑拆分为多次 LLM 调用或多 Agent 模式。

### 6.3 补赎机制未实现

宪章和架构中明确要求的补赎流程（错误后的降权、规则修订、案例写入）尚未落地。

### 6.4 回退机制未实现

架构设计了 Rollback Controller，当前系统无法对已执行的判断进行回退操作。

### 6.5 接入层的自动化能力不足

风险预分类和上下文关联仍为用户手动操作，未实现自动分类器。

## 八、总体评价

**MVP 的核心价值已验证成功。** 系统完整实现了"先净化判断，再允许执行"的核心链路，六阶段流程、三类结论、五戒检查、因果记录、人工复核都已跑通。

与设计初衷的一致性：**高度一致，无方向性偏离。**

存在的简化属于 MVP 阶段的合理取舍，不影响核心价值的验证。

## 九、后续建议（按优先级）

### P0：补齐核心缺失

1. **显式状态机**：让每个请求有可查询的当前状态，支持中途暂停和干预
2. **补赎机制**：人工复核推翻后触发规则修订建议、降权和案例写入

### P1：增强深度

3. **辩义引擎多角色拆分**：将单次 LLM 调用拆为多视角 Agent，产生真正的分歧和对抗
4. **风险自动分类**：根据请求内容自动预判风险等级，减少手动操作
5. **回退机制**：支持对已执行判断的回退操作和记录

### P2：扩展生态

6. **案例库**：积累历史案例供观缘和辩义引用
7. **审计日志**：独立的操作审计表，记录所有系统和人工操作
8. **API 集成**：Webhook 和第三方系统接入
